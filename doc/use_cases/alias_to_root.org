When you work on linked but separate projects — say a backend API and a
frontend app — changes in one often need to be verified in the other. If they
live in separate directories, you end up constantly ~cd~-ing back and forth,
losing context along the way.

clk projects scope their commands to directories: aliases, parameters and
scripts only activate when you're inside the project. That's great for
isolation, but painful when you need to jump between two or three repos in a
single workflow.

* The problem

Let's set up two projects to see the problem in action.

cd billing-api
#+END_SRC

Each project works fine on its own. But the moment you're inside ~billing-app~
and want to rebuild the API, the command isn't there.

#+NAME: api-build-from-app

You only see the frontend's ~build~. To reach the API, you'd need ~--project~.

#+NAME: verbose-project
#+BEGIN_SRC bash :results verbatim :exports both :session 55ed49b4-065d-440e-903b-658d1dcec014 :cache yes
clk --project ../billing-api build 2>/dev/null
#+END_SRC

#+RESULTS[bdb754453d2d48d4dd67b47c95baa2b62c061b1b]: verbose-project

That works, but typing ~clk --project ../billing-api~ every time you switch
context gets old fast — especially when you're going back and forth several
times in a row.

* Global aliases

This is where an alias to the root command helps. You can create a global alias
that points to ~clk~ itself, with the ~--project~ option baked in. Do this once
for each project.

#+NAME: leave-app
#+BEGIN_SRC bash :results none :exports code :session 55ed49b4-065d-440e-903b-658d1dcec014
cd ..
#+END_SRC

#+NAME: create-api-alias
#+BEGIN_SRC bash :results verbatim :exports both :session 55ed49b4-065d-440e-903b-658d1dcec014 :cache yes
clk alias set api clk --project ./billing-api
#+END_SRC

#+RESULTS[790ec28c9431c53e40d2f65fdf420f3d27ff4645]: create-api-alias
: New global alias for api: clk --project ./billing-api

#+NAME: create-app-alias
#+BEGIN_SRC bash :results verbatim :exports both :session 55ed49b4-065d-440e-903b-658d1dcec014 :cache yes
clk alias set app clk --project ./billing-app
#+END_SRC

#+RESULTS[74872c53ce36d9380c026c3eeb5c0cfc376ef446]: create-app-alias
: New global alias for app: clk --project ./billing-app

Now every command from both projects is available from anywhere, under ~api~ and
~app~. No more ~cd~-ing around.

* Local aliases

The global approach is convenient but pollutes your top-level namespace. If you
only need cross-project shortcuts when you're actually working inside one of the
projects, local aliases are a cleaner fit. Each project declares its own
shortcut to the sibling, and nothing leaks outside.

First, let's remove the global aliases.

#+NAME: remove-global-api-alias
#+BEGIN_SRC bash :results verbatim :exports both :session 55ed49b4-065d-440e-903b-658d1dcec014 :cache yes
clk alias unset api
#+END_SRC

#+RESULTS[148d12a3901956002b23843b18861630d4fa00a6]: remove-global-api-alias
: Erasing api alias from global settings

#+NAME: remove-global-app-alias
#+BEGIN_SRC bash :results verbatim :exports both :session 55ed49b4-065d-440e-903b-658d1dcec014 :cache yes
clk alias unset app
#+END_SRC

#+RESULTS[586426e2c7d48fe1eae63aa5cbaa8beadc984fa9]: remove-global-app-alias
: Erasing app alias from global settings

#+NAME: enter-api-for-local
#+BEGIN_SRC bash :results none :exports code :session 55ed49b4-065d-440e-903b-658d1dcec014
cd billing-api
#+END_SRC

#+NAME: local-api-app-alias
#+BEGIN_SRC bash :results verbatim :exports both :session 55ed49b4-065d-440e-903b-658d1dcec014 :cache yes

#+END_SRC

#+RESULTS[74c1ecba8058eeb2c267ef61f727316d7d126dee]: local-api-app-alias

#+NAME: leave-api-for-local

your global namespace or show up in unrelated directories.
