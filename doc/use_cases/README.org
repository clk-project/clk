These use cases illustrate clk features through concrete examples. They show
common patterns that emerged from real usage and give hints about when clk
might be useful.

There are organised by feature though so you can find easily what you are
looking for.

* creating commands
  :PROPERTIES:
  :CUSTOM_ID: creating-commands
  :END:
** bash commands
   :PROPERTIES:
   :CUSTOM_ID: bash-commands
   :END:

   The basics are covered [[file:bash_command.org][here]]. To get more into how to provide parameters to
   your command, read [[file:bash_command_use_option.org][that one]]. If your parameters are too complicated to be
   simply parsed, follow the idea of this [[file:send_sms.org][use case]].

   If you find out that your bash command starts to become quite big, read [[file:bash_command_import.org][this]]
   to find out how to split your code.

   Most of my bash commands start with simple aliases (see [[*aliases][aliases]]). You can
   read more about this way of thinking in [[file:bash_command_from_alias.org][here]]. The [[*standard library][standard library]] section
   covers the built-in helpers available in bash commands.

** python commands
   :PROPERTIES:
   :CUSTOM_ID: python-commands
   :END:

   The basics of creating python commands are covered [[file:python_command.org][here]]. For more advanced
   patterns involving dynamic parameters and exposed classes, see [[*dynamic parameters and exposed classes][dynamic
   parameters and exposed classes]].

** dynamic parameters and exposed classes
   :PROPERTIES:
   :CUSTOM_ID: dynamic-parameters
   :END:

   To create powerful, dynamic command line tools that provide the best
   completion possible, see [[file:dynamic_parameters_and_exposed_class.org][this pattern]] for writing your commands. In case you
   have some issues, you might want to look at the [[file:dynamic_parameters_advanced_use_cases.org][advanced use cases]].

** commands as first order objects
   :PROPERTIES:
   :CUSTOM_ID: commands-as-objects
   :END:

   Sometimes, you create commands not only to be called directly, but to be used
   as basis to build greater commands.

   This example of an [[file:ethereum_local_environment_dev_tool.org][ethereum local environment dev tool]] shows how to plug clk
   commands as parameters in other commands.

   [[file:ipfs_name_publish.org][ipfs name publish]] shows how to use clk bash commands to create the completion
   for other commands.

* aliases
  :PROPERTIES:
  :CUSTOM_ID: aliases
  :END:

  Aliases let you create shortcuts, compose commands, and inject environment
  variables.

  For an example of using aliases with templated environment variables to create
  flexible workflows, see the [[file:podcast_automation.org][podcast automation]] example.

  For shortcut aliases that target specific deployment environments, see the
  [[file:environment_deployment.org][environment deployment]] use case. It also shows how to use
  ~get_parameter_source~ to distinguish between default and explicit values.

  When a global alias and a local command share the same name, the local command
  wins -- see [[*resolution priority][resolution priority]]. You can also create aliases that [[file:alias_to_root.org][point to
  another project's root command]] for [[*cross-project access][cross-project access]].

** values (semantic defaults)
   :PROPERTIES:
   :CUSTOM_ID: values
   :END:

   When you need to set the same option on many commands, consider using [[file:setting_default_values.org][values
   to set semantic defaults]]. This explains the difference between syntactic
   (parameters) and semantic (values) configuration, with a comparison to git
   config.

** resolution priority
   :PROPERTIES:
   :CUSTOM_ID: resolution-priority
   :END:

   clk resolves names by looking at the local profile first, then the global
   profile, then aliases and other resolvers. A local custom command always wins
   over a global alias with the same name, and a global command always wins over
   an alias.

   Several use cases show this in action from different angles:

   - [[file:global_workflow_local_implementation.org][global workflow, local implementation]] -- a global ~test~ alias exists, but
     when you enter a project that defines a local ~test~ bash command, the
     local command wins. The global ~test-n-push~ workflow automatically picks
     up the local implementation.
   - [[file:wrapping_a_cloud_provider_cli.org][cloud provider CLI wrapper]] -- global parameters set a default AWS profile,
     but local parameters in a project directory override them. When you leave
     the project, the global defaults take over again. Extensions named
     ~config-*~ override global settings, so enabling a ~config-staging~
     extension [[file:wrapping_a_cloud_provider_cli.org::*switching environments with extensions][switches the whole CLI context]] without touching global
     configuration.
   - [[file:alias_to_root.org][alias to root]] -- local aliases can be scoped to a project, so they only
     shadow global names when you are inside that directory.

* projects
  :PROPERTIES:
  :CUSTOM_ID: projects
  :END:

  A project is a directory with a ~.clk~ folder. Commands, aliases, and
  parameters defined inside are scoped to that directory.

  The [[file:using_a_project.org][basics]] of setting up and using a project. See [[*resolution priority][resolution priority]] for how
  local project configuration takes precedence over global settings.

** cross-project access
   :PROPERTIES:
   :CUSTOM_ID: cross-project-access
   :END:

   When you work on linked but separate projects and get tired of ~cd~-ing
   between them, you can create [[file:alias_to_root.org][aliases that point to the root command]] of another
   project. This lets you reach any command from a sibling project without
   leaving your current directory, using either global or local aliases
   depending on how much namespace isolation you want.

* flows
  :PROPERTIES:
  :CUSTOM_ID: flows
  :END:

  Flows let you chain commands in a sequence with declared dependencies.

  clk does not want to compete with dedicated flow tools like nodered, but it
  helps having basic flow handling, like when you have a [[file:3D_printing_flow.org][3D printing flow]]. The
  [[*end-to-end example][end-to-end example]] combines flows with [[*parameters and configuration][parameters]], [[*aliases][aliases]], and [[*projects][projects]].

* standard library
  :PROPERTIES:
  :CUSTOM_ID: standard-library
  :END:

  clk tries hard to provide most of what you need in a generic command line
  tool.

  The bash library (_clk.sh) provides [[file:bash_command_built_in_lib.org][helpers]] for creating friendly shell
  commands.

  [[file:choices.org][Choices]] for providing selection in commands. [[file:scrapping_the_web.org][Caching]] computation results to
  disk. [[file:fetching_and_displaying_json_data.org][Fetching and displaying JSON data]] with ~download~ and ~echo_json~.
  Handling [[file:dealing_with_secrets.org][secrets]] in commands.

  The [[file:lib.org][clk.lib reference]] covers what has been documented so far.

* introspection and debugging
  :PROPERTIES:
  :CUSTOM_ID: introspection-and-debugging
  :END:

  The ~clk describe~ command helps you [[file:self_documentation.org][explore and document your configuration]],
  showing what [[*aliases][aliases]], [[*parameters and configuration][parameters]], commands, and [[*extensions][extensions]] are available
  in any profile (global, local, or extension). This is especially useful when
  [[*resolution priority][resolution priority]] makes it unclear which level a setting comes from.

  When a command feels slow, use [[file:spotting_slow_code.org][~--timestamp~, ~--debug~, and ~--profiling~]]
  to progressively narrow down the bottleneck, from high-level timing down to
  function-level detail.

* extensions
  :PROPERTIES:
  :CUSTOM_ID: extensions
  :END:

  To distribute your commands as installable packages, see [[file:creating_extensions.org][how to create your
  own extensions]]. Extensions can be shared across [[*projects][projects]] and follow the
  same [[*resolution priority][resolution priority]] rules. They can also be used to
  [[file:wrapping_a_cloud_provider_cli.org::*switching environments with extensions][override global parameters]], making it easy to toggle between sets of
  settings (e.g. staging vs production). For deeper customisation, see
  [[*plugins][plugins]].

* plugins
  :PROPERTIES:
  :CUSTOM_ID: plugins
  :END:

  Plugins let you dynamically monkey-patch clk internals (at your own risk).

  We [[file:using_a_plugin.org][describe here]] how a command can be added along with modifications to clk's
  invoke system. This is almost dark magic, so we don't actually expect you to
  need this, but it has been useful in a few very specific situations. For most
  sharing needs, [[*extensions][extensions]] are a better fit.
